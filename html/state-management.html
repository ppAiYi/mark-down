状态管理

依赖

  • Vue @2.x
  • Vuex @2.x
  • Flux Standard Action @1.x

引言

在组件化开发的基础上，对于一个前端系统，我们将各组件包含的属性和数据统称为系统的 “状态（state）”，其实际展示的内容统称为系统的 “视图（view）”。此时，可以把系统整体理解为一个从状态到视图的映射，即：

view = f(state)

其中 state 代表系统的状态，包括各组件对应的属性和数据；view 代表系统的视图，常见形式包括 DOM 元素片段，CSS 样式和其他静态资源等；在映射法则 f 的作用下，系统的任意一个状态都将唯一对应一个视图。

系统视图的形式并不仅限于在浏览器端运行的网页，取决于映射法则的具体实现。例如当采用 React Native 实现映射法则时，系统视图将会是 Object C 或者 Java 组件。因此在这个模型里，系统视图是没有平台限制的 —— 这种关注分离有助于我们更好地理解和开发跨平台应用。

Vuex

一个实际的前端系统可能包含数十甚至上百个状态，直接对它们进行操作是非常复杂和混乱的；同时我们需要通过对状态的读取和修改来实现所有业务逻辑，因此有必要引入一套机制来统一管理这些状态。基于目前的组件化框架，我们采用 Vue 官方提供的类 Flux 状态管理方案 Vuex @2.x。

Flux 是 Facebook 专门针对复杂 Web 应用开发所提出的系统架构，社区再此基础上实现了很多改进后的版本，例如 Redux 是当前 React 生态环境中最流行的 Flux 实现。

完整的 Vuex 方案需要实现以下内容：

  • 系统状态树

  • 管理系统状态树的中心 Store

  • 单向数据流机制

方案相关代码统一保存在项目的 /src/vuex 路径下：

|---- package.json
|---- src
   |---- vuex        # vuex 方案相关代码

系统状态树

系统状态树负责保存完整的系统状态并实现相关业务逻辑，因此除 state 外，还必须包括与之对应的 action，mutation 和 getter。

当 state 数量较多时，系统状态树可能会变得非常臃肿。对前端系统而言，页面是逻辑和代码最自然的边界，因此我们根据页面文件的目录树来设计系统状态树结构：

  • 每个文件目录树节点唯一对应一个系统状态树节点
  • 系统状态树采用扁平化结构，所有节点位于同一层级
  • 系统状态树各节点以对应的文件目录树节点的路径作为唯一标识
  • 文件目录树的叶子节点代表实际页面，对应的系统状态树节点保存该页面的专有状态
  • 文件目录树的分支节点不代表实际页面，对应的系统状态树节点保存其子节点（可能叶子节点或下一级分支节点）的共享状态

例如一个具有如下文件目录树的系统（SPA）：

|---- package.json
|---- src
   |---- pages
      |---- index.html             # 根页面文件
      |---- index.js
      |---- login
         |---- index.vue           # login 页面文件
      |---- home
         |---- main
            |---- index.vue        # home/main 页面文件
         |---- detail
            |---- index.vue        # homt/detail 页面文件

其对应的系统状态树应为：

{
    '/': { ... },                // 对应文件目录树根节点，保存系统全局状态
 
    'login': { ... },            // 对应文件目录树叶子节点，保存 login 页面的专有状态
 
    'home': { ... },             // 对应文件目录树分支节点，保存下属页面的共享状态
 
    'home/main': { ... },        // 对应文件目录树叶子节点，保存 home/main 页面的专有状态
 
    'home/detail': {... }        // 对应文件目录树叶子节点，保存 home/detail 页面的专有状态
}

Modules

我们将每个节点对应的 state，action，mutation 和 getter 整合在一个独立的模块（module）中实现。每个模块唯一对应一个工程目录和文件（index.js），统一保存在项目的 /src/vuex/modules 路径下。为使代码结构更加直观，所有模块的文件目录树与对应页面保持一致。

如上例中，系统的文件目录树应为：

|---- package.json
|---- src
   |---- pages
      |---- index.html               # 根页面文件
      |---- index.js
      |---- login
         |---- index.vue             # login 页面文件
      |---- home
         |---- main
            |---- index.vue          # home/main 页面文件
         |---- detail
            |---- index.vue          # homt/detail 页面文件
   |---- vuex
      |---- modules
         |---- index.js              # 根节点对应 module
         |---- login
            |---- index.js           # login 节点对应 module
         |---- home
            |---- index.js           # home 节点对应 module
            |---- main
               |---- index.js        # home/main 节点对应 module
            |---- detail
               |---- index.js        # home/detail 节点对应 module

此外，为保证代码一致性以及避免在状态树合并的过程中产生冲突，我们为 module 制定了一些特殊的开发规范。下面以 home/main 节点对应的 module 为例，展示模块内各部分的具体实现细节：

state

state 以对象形式保存该节点对应的所有状态和初始值。

const state = {
    userId: 0
};
mutation type

mutation type 用于唯一确定一个 mutation 的类型，需遵循以下规范：

  • 以常量形式实现，且常量名和常量值应保持一致
  • 所有字母全部大写，不同单词间以下划线 _ 分隔
  • 命名符合 节点标识 + 动作 + 状态 结构
  • 动作 必须为动词，且能够准确描述 mutation 对应的行为
  • 状态 必须与 state 中对应状态的命名保持一致
const HOME_MAIN_SET_USER_ID = 'HOME_MAIN_SET_USER_ID';
mutations

mutations 以对象形式保存各类 mutation 的响应函数，需遵循以下规范：

  • 每个 mutation type 唯一对应一个响应函数
  • 直接使用 mutation type 常量作为函数名（通过 ES6 属性名表达式）
const mutations = {
    [HOME_MAIN_SET_USER_ID](state, mutation) {
        state.userId = mutation.payload;
    }
};
actions

actions 以对象形式保存各类 mutation 的触发函数，需遵循以下规范：

  • 每个 mutation type 至少对应一个触发函数
  • 命名符合 节点标识 + 动作 + 状态 结构，采用 camelCase 拼写方式
  • 动作 必须与 mutation type 中对应动作的命名保持一致
  • 状态 必须与 mutation type 中对应状态的命名保持一致
  • 返回值必须是一个 Promise 对象，用于实现链式调用
  • 生成的 mutation 对象必须符合 Flux Standard Action @1.x 规范
const actions = {
    homeMainSetUserId({ commit }, userId) {
        return new Promise((resolve, reject) => {
            if (util.isNumber(userId)) {
                dispatch({
                    type: HOME_MAIN_SET_USER_ID,
                    payload: userId
                });
                resolve();
            } else {
                reject(new Error('invalid parameter'));
            }
        });
    }
};
getters

getters 以对象形式保存 state 对应的计算函数，需遵循以下规范：

  • 每个 mutation type 至少对应一个计算函数
  • 命名符合 节点标识 + 状态 结构，采用 camelCase 拼写方式
  • 状态 必须与 mutation type 中对应状态的命名保持一致
const getters = {
    homeMainUserId(state) {
        return state.userId;
    }
};

综上，home/main 节点对应 module 的完整实现代码如下：

import util from 'util';
 
 
 
const state = {
    userId: 0
};
 
const HOME_MAIN_SET_USER_ID = 'HOME_MAIN_SET_USER_ID';
 
const mutations = {
    [HOME_MAIN_SET_USER_ID](state, mutation) {
        state.userId = mutation.payload;
    }
};
 
const actions = {
    homeMainSetUserId({ commit }, userId) {
        return new Promise((resolve, reject) => {
            if (util.isNumber(userId)) {
                commit({
                    type: HOME_MAIN_SET_USER_ID,
                    payload: userId
                });
                resolve();
            } else {
                reject(new Error('invalid parameter'));
            }
        });
    }
};
 
const getters = {
    homeMainUserId(state) {
        return state.userId;
    }
};
 
export default {
    state,
    mutations,
    actions,
    getters
};

中心 Store

中心 Store 在项目的 /src/vuex/store/index.js 文件中实现，用于统一管理系统状态树。

|---- package.json
|---- src
   |---- vuex
      |---- modules
      |---- store
         |---- index.js        # 中心 Store

Store 将加载所有 module 并将它们合并成完整的系统状态树，对于上面例子中的系统，其中心 Store 的参考实现如下：

import Vue from 'vue';
import Vuex from 'vuex';
import rootModule from '../modules';
import loginModule from '../modules/login';
import homeMainModule from '../modules/home/main';
import homeDetailModule from '../modules/home/detail';
 
 
 
Vue.use(Vuex);
 
export default new Vuex.Store({
    modules: {
        '/': rootModule,
        'login': loginModule,
        'home/main': homeMainModule,
        'home/detail': homeDetailModule
    }
});

单向数据流

在目前的开发中，我们统一使用 Vue @2.x 作为映射法则的具体实现。为实现单向数据流机制，Vue 组件必须：

  • 不包含任何业务逻辑，仅实现简单的数据格式化处理
  • 通过加载 module 获取指定的状态树节点
  • 不直接修改系统状态，通过调用 action 要求中心 Store 执行指定的业务逻辑来更新 state，以响应用户操作
  • 通过调用 getter 从中心 Store 获取对应的 state 来更新视图

下面以 home/main 页面对应的 Vue 组件为例，展示单向数据流机制的具体实现细节。

首先在页面中放置一个按钮，点击按钮将随机生成一个 userId 并以格式化形式显示在页面上：

<template>
    <div class="container">
        {{ formattedUserId }}
        <button v-on:click="createUserId"> Click! </button>
    </div>
</template>

然后在组件中引用该页面 module 对应的 action 和 getter：

<script>
    import { mapActions, mapGetters } from 'vuex';
 
    export default {
        computed: {
            ...mapGetters([
                'homeMainUserId'
            ])
        },
        methods: {
            ...mapActions([
                'homeMainSetUserId'
            ])
        }
    };
</script>

接着实现按钮的点击事件响应方法，该方法将随机生成一个 userId，并调用 action 把它传递给中心 Store：

    createUserId() {
        const userId = Math.random();
 
        this.homeMainSetUserId(userId);
    }

最后在组件的计算属性中通过调用 getter 来实时获取相关状态的值，并对其进行格式化处理：

    formattedUserId() {
        return this.homeMainUserId * 10;
    }

综上，home/main 页面对应 Vue 组件的完整实现代码如下：

<template>
    <div class="container">
        {{ formattedUserId }}
        <button v-on:click="createUserId"> Click! </button>
    </div>
</template>
 
<script>
    import { mapActions, mapGetters } from 'vuex';
 
    export default {
        computed: {
            ...mapGetters([
                'homeMainUserId'
            ]),
            formattedUserId() {
                return this.homeMainUserId * 10;
            }
        },
        methods: {
            ...mapActions([
                'homeMainSetUserId'
            ]),
            createUserId() {
                const userId = Math.random();
 
                this.homeMainSetUserId(userId);
            }
        }
    };
</script>
